## 1. Авторизация

**Решение: Email + Пароль (Supabase Auth)**

- Регистрация: email + пароль при оформлении подписки на лендинге
- Сессия: 30 дней, автообновление в фоне (пользователь не замечает)
- "Забыл пароль": письмо со ссылкой сброса
- Без авторизации: только экран входа — контент недоступен
- Сложность реализации: низкая (Supabase из коробки)

**Варианты которые отклонили:**
- Magic Link — неудобно при истечении сессии
- Google OAuth — часть аудитории не хочет светить аккаунт для деликатной темы

## 2. Данные пользователя и обновления

**Правило: Supabase = источник правды, localStorage = только кэш**

### Таблицы Supabase

| Данные | Таблица |
|---|---|
| Профиль и подписка | `user_profiles`, `subscriptions` |
| Прогресс по дням | `user_progress` |
| Ежедневные отчёты | `reports` |
| История чата с тренером | `chat_messages` |
| Диагностики и опросы | `diagnostics` |
| Промокоды / токены доступа | `access_tokens` |

### Обновления приложения

Принудительное обновление (стиль Battle.net — без bypass):

1. При запуске приложения → запрос `version.json` на сервере
2. Сравниваем `APP_VERSION` в коде с `server_version`
3. Если `force: true` и версия устарела → полноэкранный блок "Требуется обновление"
4. Единственная кнопка: "Обновить" — обойти нельзя
5. Service Worker скачивает новый код → автоперезагрузка
6. Данные в Supabase не трогаются

Заготовки уже есть: `UpdatePrompt.tsx`, `VersionGateBanner.tsx`, `appVersion.ts`

## 3. Тестовый доступ / Промокоды

**Решение: Временные токены в таблице `access_tokens`**

Структура записи:
- `token` — уникальный код (напр. `FRISKIALPHA2026`)
- `plan` — тариф (`premium`, `basic`, `week1`)
- `duration_days` — на сколько дней активируется
- `expires_at` — до какой даты токен принимается
- `max_uses` — лимит использований
- `used_count` — сколько раз использован

Пользователь вводит токен → проверяем → активируем подписку на `duration_days`.

Используется для:
- Тестовых аккаунтов разработчика
- Бета-групп (Frisky Alpha и будущие)
- Акций и партнёрств
- Подарочных доступов

## 4. Frisky Alpha — бесплатный доступ

Участники тестового потока (Telegram mini-app) получают 30 дней бесплатно.

**Реализация:**
1. Создаём токен `FRISKIALPHA` в таблице `access_tokens`
   - plan: `premium`
   - duration_days: 30
   - max_uses: [количество участников]
   - expires_at: [дата окончания акции]
2. Отправляем им письмо с кодом и инструкцией
3. Они вводят код при регистрации → подписка активируется автоматически

## 5. Разблокировка дней (таймер)

**Два условия для открытия дня N:**
1. День N-1 завершён (отчёт заполнен и отправлен)
2. Наступило 01:00 следующего календарного дня **по локальному времени пользователя**

**Логика:**
- `course_start_date` — дата начала курса (записывается при старте)
- `day_completed_at[N]` — timestamp завершения дня N (заполнения отчёта)
- День N открывается когда: `day_completed_at[N-1]` существует И `now() >= следующий день в 01:00`

**Правила (решено 13.04.2026):**
- Пропущенный день: пока не заполнены ВСЕ поля отчёта за день N-1 — день N не открывается. Без исключений.
- Это прозрачно для пользователя: он видит что именно нужно сделать.
- Streak сбрасывается при пропуске дня (отчёт не сдан до 01:00).

### Тестовый режим для разработки
- Переменная `VITE_TEST_MODE=true` в `.env.local` → все дни открыты, таймер отключён
- Только локально, в продакшн не попадает
- Для бета-тестеров — токен с планом `test` (таймер отключён, все дни доступны)

## 6. Flow: Оплата → Вход в приложение

**Самый важный и неочевидный кусок — пока не реализован.**

Задача: пользователь оплатил на лендинге → должен попасть в PWA уже залогиненным.

**Решение:**
1. На лендинге: форма оплаты включает поле email + придумать пароль
2. При успешной оплате: создаём Supabase-аккаунт (email + пароль) + запись в `subscriptions`
3. Страница "Оплата успешна": показываем инструкцию установки PWA + кнопку "Открыть приложение"
4. В PWA: форма входа (email + пароль) — те же данные что вводили на лендинге

Альтернатива (проще): на странице успеха отправляем Magic Link на email → пользователь кликает → попадает в PWA залогиненным сразу.

**Статус:** требует реализации до запуска.

## 7. Push-уведомления

**Зачем:** ключевой фактор retention — напоминание "время практики".

**Как работает в PWA:**
- При первом запуске запрашиваем разрешение
- Service Worker отправляет локальные уведомления по расписанию
- Пользователь выбирает время напоминания (напр. 21:00)

**Что показываем:**
- Ежедневное напоминание о практике
- "Серия прервётся, если не сделаешь сегодня"
- Напоминание об истечении подписки (за 7 дней)

**Статус:** важно, можно добавить после запуска (v1.1).

## 8. Истечение подписки

**Решено 13.04.2026:**
- При истечении: полная блокировка, экран "Подписка закончилась" + кнопка "Продлить" → лендинг
- Цель: автоматическое продление подписки через Тинькофф (рекуррентные платежи)

### Автопродление подписки

**Юридически:** в России разрешено. Условие — пользователь должен явно согласиться при оформлении (чекбокс "Я согласен на автоматическое продление"). Это стандартная практика (как у кинопоиска, сбера и т.д.)

**Как работает технически (Тинькофф Рекуррент):**
1. Первый платёж: обычная оплата, но с флагом `Recurrent: Y`
2. Тинькофф сохраняет карту и возвращает `RebillId`
3. Мы сохраняем `RebillId` в Supabase рядом с подпиской
4. За 3 дня до истечения: автоматически списываем через API с этим `RebillId`
5. Если списание прошло → продлеваем `subscription_expires_at` на следующий период
6. Если не прошло (карта истекла) → уведомляем пользователя, даём 3-7 дней grace period

**Что нужно юридически:**
- В оферте: чёткое описание условий автопродления
- При оформлении: явный чекбокс (не предзаполненный)
- В личном кабинете: кнопка "Отменить автопродление"
- Email за 3 дня до списания: предупреждение о предстоящем платеже

**Статус:** реализовать после первых продаж (v1.1), на старте — ручное продление через лендинг.

## 10. Авторское право и интеллектуальная собственность

### Что защищает закон автоматически (без регистрации)

По ст. 1259 и 1261 ГК РФ ч. 4 авторское право возникает **в момент создания** — без регистрации, без знака ©. Но явное указание усиливает позицию в спорах.

**Что защищено:**

| Объект | Основание | Что это значит |
|---|---|---|
| Методика метод Журова | Ст. 1259 ГК РФ (научное/образовательное произведение) | Никто не может копировать и продавать твою методику |
| Тексты курса (30 дней) | Ст. 1259 ГК РФ (литературное произведение) | Тексты уроков, отчётов, диагностик |
| Приложение (код) | Ст. 1261 ГК РФ (программа для ЭВМ) | Код приложения |
| Аудио/видео материалы | Ст. 1259 ГК РФ | Записи, если будут |
| Название «30:30 Протокол» | Ст. 1474 ГК РФ — **нужна регистрация** как товарный знак | Защита бренда — опционально |

### Что добавляем в продукт

**В футере лендинга и в приложении:**
```
© 2026 Журов Дмитрий. Все права защищены.
Метод Журова является авторской разработкой.
Копирование и распространение без разрешения запрещено.
```

**В оферте** (уже добавлено): запрет копирования и распространения материалов.

**В приложении** (экран "О сервисе" или футер):
```
Приложение «30:30 Протокол»
© 2026 ИП Журов Дмитрий
Метод Журова — авторская разработка Журова Дмитрия.
Все материалы защищены законодательством РФ об авторском праве.
```

### Нужна ли регистрация товарного знака?

Регистрация «30:30 Протокол» или «метод Журова» как товарного знака — опционально на старте, рекомендуется при масштабировании. Стоимость ~30 000–50 000 ₽ через патентного поверенного, срок 6–12 месяцев.

На старте достаточно явного © и пункта в оферте.
