# Спека: статус «Отчёт сохранён» + синхронизация отчёта с Supabase

> Код пока **не меняем** — это решение и промпт для следующей итерации.

---

## 1. Что сейчас в коде (ответ на вопрос «реализовано ли?»)

| Ожидание | Реализовано? |
|----------|----------------|
| Отчёт сохраняется **локально** (устойчиво после перезагрузки в том же браузере) | **Да** — `markDayComplete` → `updateProgress` → `localStorage` (`pwa_v2_course_progress`). |
| После нажатия явный **статус UI** «Отчёт сохранён» (не только тост) | **Нет** — есть `showToast` и смена текста кнопки «Завершить день» → «Сохранить отчёт»; отдельной постоянной плашки нет. |
| Отчёт **сразу уходит в Supabase** при нажатии | **Нет** — в облако пишется только очередь **`sync_events`**, и то **после** ручного «Синхронизация с облаком» в **Профиле** (`flushSyncQueue`). |
| В `sync_events` лежит **полный отчёт** | **Нет** — в очередь кладётся только `{ event: 'day_complete', day }` (`courseProgress.ts` → `enqueueSync`), **без полей отчёта**. |

Итого: желаемое поведение («статус + локально + автоматически в Supabase с полезной нагрузкой») **не реализовано**; реализована только **локальная** модель + **опциональная** отправка урезанного события с профиля.

---

## 2. Целевое поведение (продукт)

1. После успешного сохранения (запись в `localStorage` не бросила ошибку) — UI переходит в состояние **«Отчёт сохранён»** (плашка или блок под кнопкой + при желании иконка), не исчезает при скролле в отличие от тоста.
2. Если пользователь **залогинен** и Supabase настроен — **в том же действии** (или сразу после, без отдельной кнопки в профиле) отправлять в Supabase **снимок дня**: номер дня + `reportData` + `completed` + `updated_at`.
3. Если **офлайн / нет сессии** — ставить задачу в очередь и показывать «Сохранено на устройстве, отправим при входе».
4. Ошибка сети — «Сохранено локально, не удалось отправить» + кнопка «Повторить».

---

## 3. Предлагаемое решение (архитектура, без правок кода)

### 3.1 UI

- Ввести явный **`savedUi`**: `'idle' | 'saving' | 'saved_local' | 'saved_cloud' | 'error'` в `DayScreen` (или вынести в маленький хук).
- После `completeCourseDay` / повторного сохранения: `savedUi = 'saved_local'`, затем попытка облака → `'saved_cloud'` или `'error'`.
- Кнопка: при `saved_cloud` показывать неактивную «Отчёт сохранён» или вторичную «Изменить отчёт»; не полагаться только на смену «Завершить день» / «Сохранить отчёт».

### 3.2 Данные в Supabase (выбрать один канонический путь)

**Вариант A (рекомендуется): таблица `daily_reports` или расширение `course_progress`**

- Одна строка на `(user_id, day)` с колонками `report jsonb`, `completed_at`, `updated_at`.
- Клиент: `upsert` с JWT (RLS: `user_id = auth.uid()`).
- Плюс: ИИ-тренер и аналитика читают отчёты без разбора `sync_events`.
- Минус: миграция + политики RLS.

**Вариант B: расширить тело `sync_events`**

- В `enqueueSync` класть `payload: { event: 'day_complete', day, report: {...} }`.
- При `flushSyncQueue` как сейчас — в `body` попадёт полный отчёт.
- Плюс: без новой таблицы.
- Минус: ETL и тренеру неудобно; много строк событий вместо одной актуальной строки на день.

**Вариант C: только `course_progress.snapshot`**

- При каждом завершении дня — `upsert` в `public.course_progress` весь `CourseProgressState` (или срез) в `snapshot`.
- Плюс: один запрос, полное восстановление.
- Минус: большой JSON, конфликты с нескольких устройств (нужна стратегия merge или `updated_at`).

Практично: **A для отчётов по дням** + позже **C** для полного бэкапа прогресса, либо сразу **C**, если не хочется новой таблицы.

### 3.3 Когда слать

- **Сразу после** локального сохранения, если `getSession()` есть — не ждать экран Профиль.
- При появлении сессии (`onAuthStateChange`) — **дренировать очередь** незавершённых `upsert` (ретраи).

### 3.4 Подписка и `user_profiles` (отдельно от отчёта)

- Остаётся инфраструктурная задача: webhook Тинькофф → `user_profiles` → триггер `subscriptions` (см. `DATA-FIX-RUNBOOK-AUTH-AND-REPORTS.md`).
- Код PWA отчёта это **не заменяет**.

## 4. Промпт для агента / разработчика (вставить в Cursor при реализации)

```
Задача: экран дня курса (Джаан PWA).

1) UI-состояние после сохранения отчёта
- Файл: Джаан/src/screens/DayScreen.tsx (и при необходимости вспомогательный хук).
- После успешного вызова completeCourseDay (и при повторном «Сохранить отчёт») показывать явный постоянный блок статуса:
  - «Отчёт сохранён на устройстве» → при успешной отправке в Supabase «Отчёт сохранён в облаке».
- Сохранить showToast по желанию, но не делать его единственной обратной связью.
- В состоянии «сохранено в облаке» кнопка не должна выглядеть как «ничего не произошло» (disabled + текст или отдельная «Изменить»).

2) Синхронизация отчёта с Supabase сразу после сохранения
- Не полагаться только на flushSyncQueue из Profile.
- Если пользователь залогинен (sb.auth.getSession() есть user) и getSupabase() не null:
  выполнить upsert в выбранную таблицу (см. ниже) с полями user_id, day, report jsonb, completed_at, updated_at.
- Если сети/сессии нет: оставить текущую локальную модель + опционально расширить sync_queue payload для последующего flush (включить report в payload), либо отдельную очередь «pendingDaySync».

3) Схема БД
- Добавить миграцию SQL в Джаан/supabase/migrations/: новая таблица public.daily_reports (user_id uuid, day int, report jsonb, completed_at timestamptz, updated_at timestamptz, primary key (user_id, day)) с RLS select/insert/update only auth.uid() = user_id.
- Либо, если выбираем вариант course_progress.snapshot только: upsert public.course_progress по user_id из полного loadProgress() + profileInitial при необходимости; документировать размер и конфликты.

4) Не ломать существующее
- Локальный ключ pwa_v2_course_progress остаётся источником правды для офлайн-first.
- enqueueSync/flushSyncQueue: либо расширить payload day_complete отчётом для обратной совместимости ETL, либо оставить и добавить параллельный upsert в daily_reports — на выбор, но без дублирования багов (один канон для «отчёт в облаке»).

5) Тесты вручную
- Залогинен: завершить день → в Supabase видна строка daily_reports (или snapshot).
- Без логина: завершить день → статус «на устройстве», после входа → автодосылка или кнопка в профиле.

Ограничения: минимальный дифф, не рефакторить несвязанное; стиль как в проекте; русские строки UI.
```
